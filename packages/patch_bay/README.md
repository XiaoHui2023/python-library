# patch_bay

## 设计特性

### 配置拓扑

PatchBay 像一块可改线的配线板：每个接入点先有自己的名字，再由配置画出消息能走的方向。对端不需要互相认识，只要接到这块板上，拓扑就能在中央统一调整。读配置时可以先看有哪些点，再看点和点之间拉了哪些线。

### 条件路由

连线不是一根永远导通的线，而是可以带门禁。数据包过来时，表达式会拿当前数据包做判断，成立才让它继续走这条线。判断失败只影响当前连线，不会阻止同一来源继续尝试其它出线。

### 数据补丁

补丁适合处理“到了这条线以后，要把某几个已知字段换成目标值”的场景。它不是随手改包的脚本，而是一组具名、可复用的字段改写表。字段必须已经存在，类型也要对得上，这样路由层只做明确的小改动，不制造形状不明的新数据。

### 多线分发

一个来源可以像分岔路口一样接出多条线。每条线独立判断自己的条件，能走就投递，不能走就跳过。这个形态适合做分流、旁路观察，也适合在迁移期间让新旧目标并排接收不同的数据。

### 监听回调

监听器负责把运行中的动作翻译成外部可见的事件：谁接上了、哪条线跳过了、哪一包已经投递、哪里失败了。默认日志监听器偏向人类扫读，先把消息走向看清楚。需要接监控或自定义展示时，也可以换成自己的监听器。

## 配置

### `jacks`

| 字段 | 说明 |
|---|---|
| `name` | 接入点名称，供连线引用；列表内唯一 |
| `address` | 接入点地址，写成 `host:port`；列表内唯一 |

### `wires`

| 字段 | 说明 |
|---|---|
| `from` | 源端在 `jacks` 里声明的 `name` |
| `to` | 目标端在 `jacks` 里声明的 `name` |
| `rule` | 可选；在 `rules` 中定义的规则 id，省略时恒为真 |
| `patchs` | 可选；要应用的补丁名称列表，按顺序套用；省略或空列表表示不改写数据 |

### `patchs`

| 字段 | 说明 |
|---|---|
| `name` | 补丁名称，供 `wires.patchs` 引用 |
| `patch` | 字段改写表；键为数据包顶层字段名，值为新的目标值；字段必须已存在，且目标值类型必须与原值一致 |

示例（YAML 片段）：

```yaml
patchs:
  - name: 这条补丁的名字
    patch:
      参数名1: 要改成什么值
      参数名2: 要改成什么值

wires:
  - from: a
    to: b
    rule: some_rule
    patchs:
      - 这条补丁的名字
```

### `rules`

| 项 | 说明 |
|---|---|
| 键 | 规则 id，供 `wires` 中 `rule` 引用 |
| 值 | 表达式字符串；以当前数据包对象为根，顶层字段可直接用 `{字段名}` 引用 |
| 转发条件 | 仅表达式成立时沿该边发出；不成立或求值失败则不发出 |

示例（YAML 片段）：

```yaml
rules:
  only_message: '{type} == "message"'
  only_user_123: '{user_id} == 123'
```

### `listen`

| 字段 | 说明 |
|---|---|
| `listen` | 保留配置，默认 `8765`；常规配置可省略 |
